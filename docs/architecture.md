# MRTW Architecture

MRTW is split into a shared Core library, a WPF analyst UI, and a CLI entry point.

- `MRTW.Core` owns domain models, static analysis, runtime collection, snapshot diffing, and export packaging.
- `MRTW.App` presents the timeline workbench shown in the reference images.
- `MRTW.Cli` exposes `version`, `doctor`, `static`, `run`, `export`, `batch`, `selftest`, `list`, and `open`.

Static analysis keeps PE parsing unchanged and appends a nullable `NonPeTriage` model for LNK, script, MSI/CFBF, ZIP, OOXML, and legacy Office targets. The model is serialized by the same JSON/SQLite/HTML/Privacy export path. Non-PE triage is read-only: no target is launched, resolved, decoded, expanded to disk, or passed to COM/MSI/Office APIs. ZIP child inspection remains in memory and applies entry-count, byte-size, compression-ratio, traversal, and nested-container bounds. GUI Start and CLI `run`/`batch` retain their EXE/DLL restriction.

`CommandNormalizationService` is a pure bounded text transform. It accepts only a small Base64 command syntax and LOLBin allowlist, preserves original text and outcome, and links output to the captured process GUID, PID, timestamp, and event ID. It has no execution or I/O capability. `CaseData.NormalizedCommands` is appended as an optional collection for backward-compatible case loading.

CLI `static --privacy-mode on` passes the privacy setting into the static-export path before any static metadata, triage CSV/JSON, HTML, SQLite, or ZIP artifact is written. The shared HTML report renders an HTML-encoded **Initial Access Triage** section only when `NonPeTriage` exists.

`AnalysisOrchestrator` is the common GUI/CLI acquisition path. It establishes a shared case ID and UTC epoch, applies fail-closed Windows Firewall containment, arms ETW and waits for its ready barrier before target launch, then runs the managed runtime collector. As soon as Runtime obtains the root PID it binds ETW before publishing the live Process Start callback; ETW follows only live descendant PIDs. Before PID bind, ETW-derived structured events, network sessions, and callbacks are discarded. If a prepared capture stops, fails, or is cancelled before bind, its raw ETL is not returned as evidence and is removed only from trusted case scratch; the outcome is recorded in collection quality. Exited child PIDs are removed after their Process Exit record so PID reuse cannot admit unrelated data; the root remains tracked until capture ends. The orchestrator merges telemetry, reruns behavior correlation, and records per-collector quality metadata. Pre-launch raw kernel ETL can temporarily observe system-wide activity, so it is documented separately from the target-filtered structured output.

Runtime collection uses standard .NET process execution, file snapshots, HKCU Run/RunOnce registry snapshots, and system TCP snapshots. Snapshot collection also captures four read-only persistence surfaces before and after execution: Startup folders, Task Scheduler through its COM API, Windows service registration, and WMI permanent subscriptions through WMI COM. WMI snapshots resolve normalized `__FilterToConsumerBinding` references and emit only established Filter/Consumer bindings; orphan filters and consumers are not persistence evidence. Entries are normalized by surface and identity; only created, modified, and deleted entries become timeline, artifact, and behavior evidence. Each surface has a 512-entry/2-second bound: a detected 513th entry produces `degraded` quality with retained/dropped counts and `reason=entry-limit`, while only the first 512 are retained. Each surface independently reports available, access-denied, unavailable, or timeout quality. Timeout/cancel uses a linked cancellation token and does not retain partial results. Task XML and WMI script bodies are never requested, stored, or executed; Active Script consumers retain a script hash only. `case.sqlite` is written through `Microsoft.Data.Sqlite`, so no external `sqlite3.exe` is required. UTC event timestamps and collection quality are preserved across JSON and SQLite round-trips.

Host-security snapshots use an injectable read-only provider to compare hosts, WinINet/WinHTTP proxy, Explorer, Defender, Firewall, and Security Center values. They never invoke PowerShell, netsh, Defender CLI, service control, or a setter API. Each surface has the same 2-second/512-entry bound and reports healthy, access-denied, unavailable, timeout, canceled, or degraded quality; cancelled and timed-out reads retain no partial result.
Modified entries retain an append-only old/new pair; the timeline raw JSON carries the surface, identity, and both normalized values, which are redacted by Privacy Mode before portable export. Binary registry values are represented only as bounded byte length and SHA-256, never as an implementation type string.

`MRTW.Collectors.Etw` contains the TraceEvent-backed collector for Process, ImageLoad, TCP, and DNS events. The CLI exposes `mrtw etw-smoke` for permission and provider checks without requiring a full case run.

Native API hooks under `src/MRTW.Native` use MinHook and emit JSONL through a named pipe. Adapter installation status and an aggregate installation summary are emitted at startup; the managed pipe server also reports received lines, parse failures, and connection failures. Child processes are created suspended by the hook, instrumented, and resumed unless the caller originally requested a suspended process.

Case management is centralized in `CaseService`, and CLI/WPF both use the same JSON/SQLite/export model. Privacy-mode exports are handled by `PrivacyRedactor`.
